The Problem

Most POS software assumes a stable connection. Most stores don't have one.

A sari-sari store or carinderia in the Philippines can lose internet for minutes or hours without warning. A POS system that depends on a live API call for every sale doesn't degrade gracefully in that situation — it stops the till. On top of that, the same software is usually built around one business shape (retail shelves, or restaurant tables) and bolts the other on awkwardly, and BIR compliance fields end up hardcoded instead of configured.

Connectivity is not optional infrastructure

If checkout requires a round trip to a server, a flaky connection becomes a business outage, not a UX inconvenience.

Inventory drifts without batch and location tracking

A single "quantity" column can't represent stock split across a warehouse, a counter freezer, and a display rack, each with its own expiry.

Configuration gets hardcoded business by business

VAT rate, currency, and feature flags end up as `if (businessId === 'x')` branches instead of data, which doesn't scale past a handful of clients.

Branch management is bolted on, not designed in

Multi-branch businesses need shared catalogs with isolated inventory and per-branch BIR registration — not a `branchId` column sprinkled in after the fact.

Why This Project Exists

Built for the businesses that won't fit a generic POS template.

START POS started from a simple constraint: it has to keep working at the counter when the connection doesn't. Everything downstream follows from that. The data layer had to be local-first rather than server-authoritative on every write. The schema had to model tax, configuration, and compliance as data instead of code, because a restaurant in Quezon City, a sari-sari store in Bayombong, and a future client in Singapore all need different VAT rules without a different codebase.

The result is one Prisma schema and one React application that can run a kitchen order queue for a restaurant, a barcode-driven quick checkout for a grocery, and a variant-heavy catalog for a retail shop — switched on per branch through configuration, not forked through separate builds.

Feature Showcase

Ten modules, one schema, three business types.

Inventory, tracked per branch, per location, per batch

Stock isn't a single number on a product. Every Inventory row is scoped to a branch and an optional Location (warehouse, counter freezer, display rack), with an optional batch number and expiry date.

  • › Unlimited storage locations per branch, each trackable independently
  • › FIFO, LIFO, weighted-average, or standard costing, set at the branch level
  • › A full InventoryMovement ledger — IN, OUT, ADJUST, WASTE, and internal/external transfers — so every quantity change is auditable, not just the current total

A checkout flow that supports split payments and refunds natively

A Transaction can carry multiple Payment rows — part cash, part GCash, on the same bill — and refunds are modeled as a separate transaction type linked back to the original, not a destructive edit.

  • › Cash, e-wallet, card, and tab-style credit, mixable on one bill
  • › Vendor sessions track opening/closing cash for end-of-shift reconciliation
  • › Every line item snapshots its unit cost at sale time, so historical margin reporting survives later price changes

Purchasing feeds inventory and cost data automatically

A Purchase linked to a Supplier creates PurchaseItem rows that, on save, increase inventory and update cost prices — so cost-of-goods figures stay current without a separate reconciliation step.

  • › Supplier directory with TIN tracking for cost-side audit compliance
  • › Purchases can originate from an approved Operational Task, keeping spend accountable
  • › Per-line unit and quantity, decoupled from the product's sales unit

Tables, tabs, and recipes for food-service businesses

When ENABLE_ORDER_TAB is on for a branch, cashiers can open an order before payment so the kitchen starts preparing while the customer keeps ordering. Recipes link raw-material variants to a finished item, auto-deducting ingredient stock per sale.

  • › Order lifecycle: Pending → Preparing → Served / Cancelled
  • › Add-ons are just ProductVariant rows flagged as add-ons — extra rice and an extra espresso shot use the same model
  • › Recipe costing auto-computes margin so a loss-making dish surfaces before it becomes a pattern

Reports built on the same revenue/cost split as every transaction

Because every Transaction already stores totalAmount and totalCost, the sales report's revenue-vs-cost graph is a query, not a separate calculation engine.

  • › Revenue vs. cost vs. gross profit, charted over any date range
  • › Tax summary broken into VATable, zero-rated, exempt, and VAT collected — formatted for BIR submission
  • › Inventory valuation by FIFO/LIFO/weighted-average, and an expiry tracker for perishables

Operational tasks replace verbal instructions with an audit trail

A shelf refill, a stock count, a cash reconciliation — anything that moves inventory or money outside a normal sale — goes through an OperationalTask with five accountability roles: creator, approver, clerk, reviewer, and canceler.

  • › Seven-state lifecycle, including a terminal Cancelled state with a recorded reason
  • › A flexible JSON metadata field carries task-specific context without a schema migration per task type
  • › Purchases, vendor sessions, and inventory movements can all originate from an approved task

Notifications that route by business, branch, and priority

A low-stock alert, a new order, a compliance reminder — each is scoped to a user and a business, and optionally to a branch, so a cashier at Branch A never sees an alert meant for Branch B.

  • › Five priority levels, from low to urgent
  • › Five typed notification kinds — low stock, new order, system alert, task assigned, compliance reminder — instead of a free-text message field
  • › A deep link field routes a tapped notification straight to the relevant record

Settings as data, scoped three ways

Twelve configuration keys — VAT rate, currency, low-stock threshold, feature toggles, and more — can each be set at the business, branch, or user level via a single SystemConfig table, instead of one settings column per concern.

  • › Five feature flags (order tab, kitchen queue, cash reconciliation, tasks, auto-print) turn modules on per branch without a different build
  • › BIR identifiers (TIN, PTU number, PTU issue date) live in a separate ComplianceRegistry, not mixed into operational config
  • › A PH branch and a future SG branch can run different VAT/GST rules under the same account

Customer records that scale from walk-ins to corporate accounts

Most sales don't need a customer record at all — customerId is optional on every transaction. When one is needed, billing fields like tax ID and address are stored for invoice-grade documentation.

  • › Optional customer linkage keeps fast walk-in checkout fast
  • › Buyer name, tax ID, and address can also be captured directly on the transaction for one-off corporate sales without creating a customer record
  • › Soft-deleted, not erased — a closed customer account doesn't break historical reporting

A supplier directory that exists for audit, not address-book convenience

Each Supplier carries a tax ID so the cost side of a transaction is as auditable as the revenue side — relevant the moment a BIR examiner asks where a purchase came from.

  • › Unique per business, so two tenants can both have a supplier named the same thing without collision
  • › Every purchase links back to its supplier for spend-by-vendor reporting
  • › Soft-deleted to preserve historical purchase records if a relationship ends

Offline-First

The till keeps ringing. The network catches up later.

Every write — a new sale, a stock adjustment, a task update — lands in a local SQLite store in the browser first. The UI updates immediately from that local store. A background sync layer pushes queued writes to Postgres through Prisma whenever a connection is available, and resolves the local view against the server's response.

Optimistic by default

TanStack DB applies writes to the local collection immediately and reconciles with the server response when it arrives, instead of blocking the UI on a round trip.

Background sync, not manual refresh

Queued mutations flush automatically when connectivity returns, via TanStack Query's mutation layer talking to TanStack Start server functions.

Conflict resolution is deliberately scoped down

Today, last-write-wins is acceptable because a single device per session is the common case. A CRDT-style or operation-log approach for true multi-device conflict resolution is on the roadmap, not pretended to be solved already.

A service worker for the shell, a database for the data

Serwist caches the application shell so the POS screen itself loads offline; TanStack DB's SQLite persistence is what keeps business data usable while disconnected.

Architecture

Eight layers, one direction of trust.

Reads and writes flow down from the UI to Postgres, and confirmations flow back up. Each layer has exactly one job, which is what makes the offline behavior tractable instead of a tangle of special cases.

Why TanStack DB instead of plain TanStack Query

Query is built around request/response caching. The POS needed live, queryable collections that work the same way whether the network is present or not — TanStack DB's collection model fits that shape; ad-hoc cache invalidation doesn't.

Why TanStack Start instead of a separate API server

Server functions colocated with routes removed an entire API-contract layer (REST or GraphQL schemas) for a single-frontend product, in exchange for tighter coupling to the React Start runtime — an acceptable tradeoff here, not a universal one.

Engineering Challenges

The decisions that were actually hard, and what they cost.

Offline synchronization

Optimistic local writes mean the UI is always responsive, but it means accepting that the local view can briefly disagree with the server. The tradeoff accepted today: last-write-wins reconciliation, with proper multi-device conflict resolution scoped as future work rather than solved prematurely.

Inventory consistency

Quantity isn't stored once — it's derived from a full movement ledger per variant, per location, per batch. That's more writes per sale, but it's the only way to answer "why is this number what it is" during a stock audit.

Tax abstraction

PH VAT, SG GST, and US-style sales tax are different enough that hardcoding "VAT" everywhere would have blocked expansion. Three separate enums — TaxCategory, TaxCalculationType, and TaxLineType — keep the tax logic generic at the cost of one more layer of indirection to read.

Configurability without code branches

Twelve config keys across three scopes (business, branch, user) replace what would otherwise be a settings column per concern. The cost is an extra join to resolve effective settings; the benefit is that adding a new toggle is a data change, not a deploy.

Scalability of multi-tenancy

Almost every high-traffic table is composite-indexed on businessId, branchId (and often a third column like createdAt or variantId), because nearly every query in this system is scoped to a tenant first.

Security & developer experience

Role and branch membership are checked on every route and server action. Sessions log IP and user agent for audit. Soft deletes (21 of 31 models) preserve history for BIR compliance. Biome and Lefthook enforce formatting and lint rules before a commit lands, and Zod plus Prisma's generated types close the gap between database and UI.

Tech Stack

What's actually in package.json, and why it's there.

React 19 + TanStack Router

File-based, fully type-safe routing with no string-typed paths. Tradeoff: a smaller ecosystem and fewer Stack Overflow answers than Next.js.

TanStack Start

Full-stack React SSR with server functions colocated alongside routes, removing the need for a separate API layer in a single-frontend product.

TanStack DB + SQLite persistence

Reactive client-side collections backed by an in-browser SQLite store — the engine that makes "works offline" an architectural property, not a UI nicety.

Prisma 7 (pg adapter)

Type-safe schema and migrations for 31 models. The driver adapter keeps the connection pool explicit instead of implicit, which matters under serverless deployment.

Better Auth

Session-based, role-aware authentication with IP and user-agent logging baked in, instead of a hand-rolled auth layer.

Serwist

Service-worker tooling that caches the application shell so the POS screen itself still loads when the network is down — complementary to, not a substitute for, TanStack DB.

React PDF Renderer

Generates pixel-accurate BIR-compliant receipts and invoices as real PDFs, not a styled HTML page sent to a printer.

Zod + neverthrow

Runtime validation at every server boundary, paired with a Result type instead of throwing — failure paths are explicit in the type signature, not discovered at runtime.

Tailwind CSS v4 + Shadcn/UI

Utility-first styling with accessible, unstyled Radix primitives underneath the visible components — dark-mode-first to match how the POS is actually used, on a counter, often in low light.

Biome + Lefthook

A single fast linter/formatter replacing ESLint and Prettier, with Lefthook enforcing it as a pre-commit gate rather than a CI-only check.

Vitest + Playwright

Unit/integration coverage with Vitest, end-to-end coverage with Playwright — separate tools because they're answering separate questions.

Roadmap

What's shipped, what's in progress, what's next.

Completed

  • Multi-tenant core: businesses, branches, role-scoped memberships
  • Full POS, inventory, purchasing, and operational-task workflows
  • BIR-compliant receipts, VAT/SC-PWD handling, sequential OR numbering
  • Offline-first sync v1 with optimistic local writes

In Progress

  • Hardening the sync queue for longer offline windows and larger batch sizes
  • Expanding automated end-to-end coverage of the checkout and task flows

Future

  • True multi-device conflict resolution (operation log or CRDT-based merge)
  • Multi-currency support for SG/GST branches operating alongside PH branches
  • Deeper analytics: cohort-style customer reporting, supplier spend trends

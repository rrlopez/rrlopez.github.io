1. The problem — why modal state is hard

The standard React modal pattern is a boolean in local state, passed down as a prop. It works for one dialog. It starts costing you every time you need to open a modal from an API error handler, a form submit callback, or a utility function three levels outside the component tree. 

// The pattern everyone starts with
const [isOpen, setIsOpen] = useState(false)

return (
  <>
    <Button onClick={() => setIsOpen(true)}>Open</Button>
    <ConfirmDialog open={isOpen} onClose={() => setIsOpen(false)} />
  </>
)

The moment you want to fire a warning dialog after a server mutation fails, or a success confirmation after an async workflow completes, you're reaching for context or a Zustand store — just to manage a boolean.

"I want to call showModal(WarningPrompt, { title: '...' }) from anywhere and have it just work."

That's what the overlay system delivers. Mount one component at your app root. Call showModal() everywhere else — from event handlers, server callbacks, route loaders, anywhere.

2. Installation & setup

npm install @rrlopez/overlay-manager
# or
yarn add @rrlopez/overlay-manager

Mount <Overlay /> once at your app root — the shell component that's always in the tree. In a TanStack Start project this lives in __root.tsx: 

import Overlay from '@/lib/overlay'
import { Toaster } from 'sonner'

function RootDocument({ children }) {
  return (
    <html>
      <body>
        <ThemeProvider>
          <Overlay />          // ← mounts the singleton, renders the modal stack
          {children}
          <Toaster />          // ← separate system for toasts (sonner)
        </ThemeProvider>
      </body>
    </html>
  )
}

Note that <Overlay /> and <Toaster /> are two separate systems with different jobs. Overlay handles dialogs, drawers, and confirmation prompts that need structured open/close control. Toaster (sonner) handles ephemeral notifications that auto-dismiss. Don't conflate them.

3. Basic usage: showModal() from anywhere

showModal is an async function that returns a Promise<string> — the ID of the dialog that was opened. You rarely need the return value, but it's there if you want to target that specific dialog later with delModal(id). 

import { showModal } from '@rrlopez/overlay-manager'
import { WarningPrompt } from './prompts/warning-prompt'

// Inside any component, handler, or async function:
async function handleDelete(id: string) {
  await showModal(WarningPrompt, {
    title: 'Delete ticket?',
    description: 'This cannot be undone.',
    btnText: 'Delete',
    onConfirm: async () => {
      await deleteTicket(id)
      return true  // returning true signals the prompt to auto-close
    },
  })
}

Your component receives open and onClose injected automatically. Everything else in the options object is spread onto the component as props. The only reserved key is key — for deduplication (section 6).

The full public API

showModal(Component, options?) → Promise<string>
delModal(id: string) → string
clearModals() → void
delChildModals(id: string) → void

4. How it works — the singleton class

The entire system is a single React class component called OverlayManager. Its constructor assigns OverlayManager.instance = this, making the live React component instance reachable from anywhere in the module without React context: 

class OverlayManager extends Component<Props, State> {
  static instance: OverlayManager | null = null

  constructor(props) {
    super(props)
    this.state = { overlays: [], isMounted: false }

    if (OverlayManager.instance) {
      // Hot-reload / StrictMode: clear old state and re-register
      OverlayManager.instance.clear()
      OverlayManager.instance.updateProps(props)
      return OverlayManager.instance
    }

    OverlayManager.instance = this
  }
}

The constructor also handles hot-reload and React StrictMode double-invocation gracefully: if an instance already exists it clears its state, updates its props, and returns the existing instance rather than creating a second one.

showModal is an async exported wrapper that validates the instance exists before delegating: 

export const addOverlay = async (
  ...args: Parameters<OverlayManager['addOverlay']>
): Promise<number> =>
  OverlayManager.instance.addOverlay(...args)

// The named alias used throughout the codebase:
export const showModal = addOverlay

The render method maps over this.state.overlays and wraps each open dialog in a SafeDialogGuard. Closed entries are returned as null — they stay in state for keyed deduplication but are never in the DOM: 

render() {
  return this.state.overlays.map(({ Component, open, props, id }) => (
    <Component
      key={id}
      {...props}
      open={open}
      onClose={() => this.delOverlay(id)}
    />
  ))
}

5. Collision-free IDs with ES6 generators

Each overlay needs a stable, unique ID so React can reconcile the list and so delModal(id) can target the right entry. Rather than reaching for an external UUID library, the implementation uses a plain getRandomNumber helper — the lightest possible approach for a session-scoped ID: 

function getRandomNumber(min: number, max: number): number {
  return Math.floor(Math.random() * (max - min + 1)) + min
}

// Inside addOverlay — called on every showModal():
addOverlay(Component, options = {}) {
  const id = getRandomNumber(1, 100)
  ...
}

For sessions that open a modest number of overlays this is completely collision-free in practice. The range of 1–100 means at most 100 concurrent entries before a collision becomes likely — that's a comfortable ceiling for typical UI use.

The ID is a number throughout — used as the React key, as the argument to delOverlay(id), and returned as the resolved value of the addOverlay promise. There's no external dependency, no import, no build overhead.

6. Keyed modals — one instance per intent

By default every showModal() call pushes a new entry. Pass a key to deduplicate — useful for "session expired" banners or any dialog that should only exist once at a time regardless of how many times the trigger fires: 

// Without key: two separate overlays
showModal(SessionExpired)
showModal(SessionExpired)  // ← opens a second one

// With key: second call finds and re-opens the existing entry
showModal(SessionExpired, { key: 'session-expired' })
showModal(SessionExpired, { key: 'session-expired' })  // ← still just one

The deduplication logic inside addOverlay(). Note the filter that runs first — it prunes closed keyless entries on each call, keeping the state array from growing forever:

addOverlay(Component, options = {}) {
  const id = getRandomNumber(1, 100)
  const { key, ...props } = options  // key is reserved, rest become component props

  // Discard closed keyless entries; keep open ones and all keyed ones
  const overlays = this.state.overlays.filter(({ key, open }) => open || key)

  let overlayIndex = overlays.findIndex(o => o.key && o.key === key)

  if (overlayIndex > -1) {
    // Splice out the existing entry, update it, push to end (front of render stack)
    const overlay = overlays.splice(overlayIndex, 1)[0]
    overlays.push({ ...overlay, open: true, props })
  } else {
    overlays.push({ open: true, props, Component, key, id })
  }

  this.setState({ overlays })
  return id
}

When a keyed entry is found, it's spliced out and pushed to the end of the array. Because React renders list items in order, the last item renders on top — so re-opening a keyed overlay brings it to the front of the stack automatically.

7. Closing animations & the two-step close

When a user dismisses an overlay — or the browser back button fires — you don't want the component to disappear instantly. Most modal primitives (Radix Dialog, Base UI) play an exit animation when open transitions to false, and they need to stay mounted long enough to finish it.

If you called overlays.filter(o => o.id !== id) immediately, the component would unmount before the animation runs. Instead the library uses a two-step close — flip open to false, let the primitive animate out, then the primitive removes itself from the DOM: 

// Step 1 — set open: false. Component stays mounted; animation runs.
delDueToBackButton(id: number) {
  this.setState(state => {
    const overlay = state.overlays.find(o => o.id === id)
    if (overlay) overlay.open = false
    return { ...state }
  })
  return id
}

// Step 2 — delOverlay delegates to the same method.
// The overlay primitive (Radix/Base UI) is responsible for
// removing its own DOM node once the animation completes.
delOverlay(id: number) {
  return this.delDueToBackButton(id)
}

After the flip, the entry stays in this.state.overlays with open: false. It won't be rendered because the render method only mounts entries where open is truthy — but it lingers in state until the next addOverlay call's filter sweep removes it (or clearModals() is called explicitly).

Why the primitive handles its own DOM removal

Radix Dialog, Base UI Dialog, and similar components manage their own unmount after open goes false — they wait for the CSS transition to finish, then detach. The overlay manager doesn't need to know about animation duration. It just flips the prop and steps back.

8. SafeDialogGuard — anti-tamper & remount

In a production app running in a browser the user controls, nothing stops someone from opening DevTools and removing a dialog node from the DOM while the manager still thinks it's open. SafeDialogGuard is a thin wrapper component that watches for exactly that using a MutationObserver: 

function SafeDialogGuard({ id, children, onTamper }) {
  const containerRef = useRef<HTMLDivElement>(null)

  useEffect(() => {
    const observer = new MutationObserver(mutations => {
      for (const mutation of mutations) {

        // 1. Guard container or modal element removed from the DOM
        if (mutation.removedNodes.length > 0) {
          const wasRemoved = Array.from(mutation.removedNodes).some(node => {
            if (!(node instanceof HTMLElement)) return false
            return node.id === `guard-${id}` || node.id === id
              || node.contains(document.getElementById(id))
          })
          if (wasRemoved) { onTamper(); break }
        }

        // 2. Guard container hidden via inline style (display:none / opacity:0)
        if (mutation.type === 'attributes' && mutation.attributeName === 'style') {
          const target = mutation.target as HTMLElement
          if (target.id === `guard-${id}` || target.id === id) {
            const s = window.getComputedStyle(target)
            if (s.display === 'none' || s.visibility === 'hidden'
                || parseFloat(s.opacity) === 0) {
              onTamper(); break
            }
          }
        }
      }
    })

    observer.observe(document.body, {
      childList: true, subtree: true,
      attributes: true, attributeFilter: ['style', 'class'],
    })
    return () => observer.disconnect()
  }, [id, onTamper])

  return <div id={`guard-${id}`} ref={containerRef} style={{ display: 'contents' }}>{children}</div>
}

When a tamper is detected, onTamper calls forceRemountDialog(id) on the manager. That method gives the entry a slightly mutated ID, which breaks React's reconciliation cache and forces a clean remount: 

forceRemountDialog(id: string) {
  this.setState(state => {
    const updated = state.dialogs.map(dialog => {
      if (dialog.id === id) {
        // Vary the first segment to bust React's key cache
        return { ...dialog, id: `${id.split('-')[0]}-${uuid().split('-')[0]}` }
      }
      return dialog
    })
    return { dialogs: updated }
  })
}

The global click and keydown listeners in componentDidMount are the second line of defence — a fallback for the case where the MutationObserver fires too late or misses the tamper. On any user interaction, the manager scans all open dialogs for missing DOM nodes and remounts them: 

handleGlobalInteraction = () => {
  const activeDialogs = this.state.dialogs.filter(d => d.open)
  for (const dialog of activeDialogs) {
    const domNode = document.getElementById(`guard-${dialog.id}`)
    if (!domNode) {
      console.warn(`[Overlay Guard] Shell missing. Remounting ${dialog.id}`)
      this.forceRemountDialog(dialog.id)
    }
  }
}

What SafeDialogGuard catches

  • › removeChild on the guard container or its inner modal element
  • › Inline display: none, visibility: hidden, or opacity: 0 applied directly
  • › Any DOM removal that contains the guard's child node

What it doesn't cover

  • › CSS class-based hiding (e.g. hidden utility class) — only inline style is observed
  • › Programmatic z-index changes that visually bury the dialog

9. delChildModals & clearModals

Beyond showModal and delModal, two more exports cover bulk-close scenarios.

clearModals()

Closes every overlay in the stack at once. Useful on route navigation, logout, or session expiry — any moment where you want to guarantee the screen is clean: 

clear(): void {
  const overlays = this.state.overlays.map(overlay => ({
    ...overlay,
    open: false,
    key: undefined,  // clear keys too so nothing can be re-opened
  }))
  this.setState({ overlays })
}

export const clearModals = (): void => OverlayManager.instance?.clear()

delChildModals(id)

Closes every overlay that was opened after a given one. The use case: a multi-step flow where step 1 opens dialog A, A opens dialog B, B opens dialog C. If the user cancels at A, you want B and C to close without knowing they exist: 

deleteChildOverlay(id: number): void {
  const startIndex = this.state.overlays.findIndex(o => id === o.id)

  const overlays = this.state.overlays.map((overlay, i) => {
    if (i > startIndex) return { ...overlay, open: false, key: undefined }
    return overlay
  })

  this.setState({ overlays })
}

export const delChildModals = (id: number): void =>
  OverlayManager.instance?.deleteChildOverlay(id)

The pattern works because overlays are stored in push order — later entries always have higher array indexes than earlier ones. delChildModals leverages that ordering guarantee to close a slice of the stack without a separate parent–child relationship in the data model.

10. Building a modal component that works with it

Any component passed to showModal must accept open and onClose — the two props the manager injects. Everything else is up to you. Here's the real-world WarningPrompt shape used in NUVELCO CCA: 

import type { OverlayProps } from '@rrlopez/overlay-manager'

// 1. Define your prop schema with defaults
const schema = {
  title:       'Are you sure?' as ReactNode,
  description: 'This action might have unintended consequences.' as ReactNode,
  btnText:     'Confirm' as string,
  onConfirm:   (() => Promise.resolve(true)) as () => Promise<boolean>,
}

// 2. Props = OverlayProps (open + onClose) + your schema as optional overrides
export type WarningPromptProps = OverlayProps & Partial<typeof schema>

export function WarningPrompt(props: WarningPromptProps) {
  const { open, onClose, onConfirm, title, description, btnText }
    = { ...schema, ...props }  // merge defaults

  const handleConfirm = async () => {
    if (await onConfirm()) onClose()  // auto-close on success
  }

  return (
    <Dialog open={open}>
      <DialogContent
        onEscapeKeyDown={e => e.preventDefault()}    // block escape dismiss
        onInteractOutside={e => e.preventDefault()}  // block backdrop dismiss
      >
        <DialogTitle>{title}</DialogTitle>
        <DialogDescription>{description}</DialogDescription>
        <Button onClick={onClose}>Cancel</Button>
        <Button onClick={handleConfirm}>{btnText}</Button>
      </DialogContent>
    </Dialog>
  )
}

Three things worth noting from this real implementation:

  • ›

    No onOpenChange on Dialog.

    The Radix Dialog is controlled purely through the open prop. Closing is always initiated via the manager's onClose(), never by the dialog's own interaction handlers. Escape and backdrop click are explicitly blocked.

  • ›

    onConfirm returns a boolean.

    true means "I'm done, close me." false means "keep me open" — useful for validation failures where you want to keep the dialog visible.

  • ›

    All props are optional with sensible defaults.

    You can fire showModal(WarningPrompt) with no options and get a functional dialog. Override only what you need.

The four built-in prompt components in NUVELCO CCA — AlertPrompt, WarningPrompt, SuccessPrompt, and LoadingPrompt — all follow this exact same shape, differing only in icon, color, and whether they have a confirm callback.

11. When to use it — and when not to

Good fit

  • › Confirmation dialogs triggered from form submit handlers, API error callbacks, or route loaders — anywhere React context isn't naturally available
  • › Stacked multi-step flows where each step opens the next dialog and can close all subsequent ones with delChildModals
  • › Internal tools and admin dashboards where you're iterating quickly — no boilerplate state wiring per dialog
  • › Apps already using Radix, shadcn/ui, or Base UI — the manager owns open, the primitive owns its animation

Not a good fit

  • › Dialogs that need to return a value to the caller synchronously — the async pattern works but a promise-based imperative API (like dialog.showModal()) is cleaner for that case
  • › SSR environments without careful hydration handling — the static class field persists across requests in a Node.js server context if module caching isn't isolated
  • › Teams that want full traceability in React DevTools — global singleton state is invisible to the component tree inspector

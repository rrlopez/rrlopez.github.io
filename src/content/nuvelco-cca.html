The Problem

Call center agents lose context and work when a connection drops mid-call.

A call center agent filling in a ticket during an active call can't afford for the form to freeze, fail silently, or lose data the moment the network hiccups. Most internal tooling is built with a live-API-per-keystroke assumption — which works fine on a stable corporate LAN but fails in the edge cases that matter most: a slow VPN, a brief ISP outage, or a tab reload mid-session.

Network interruptions kill in-progress ticket work

An agent mid-call who loses connectivity shouldn't also lose the partially filled ticket — the customer is still on the line.

Role boundaries need to be enforced at the data layer, not just the UI

Agents, supervisors, and admins see different queues and actions — hiding buttons isn't enough if the server functions don't enforce the same rules.

Ticket intake is only half the workflow

Creating a ticket is table stakes. Agents also need to look up history, reassign work, and track resolution state — all in one focused UI, not three tabs.

Auth state can't survive a page reload in a serverless-first world

Session data stored only on the server means a reload while offline = logged out. Local-first auth persistence keeps agents signed in and working.

Feature Showcase

Ticket intake, role dashboards, and offline sync — in one portal.

Structured ticket intake that writes locally first

When an agent submits a ticket, it lands in the local TanStack DB collection immediately — the UI confirms the write without waiting for a server round trip. The sync layer queues the mutation and flushes it to Postgres as soon as a connection is available.

  • › Validated intake form with Zod-backed server functions — bad data never reaches the database
  • › Tickets are immediately visible in the agent's queue before the server confirms
  • › Optimistic UI ensures no visible loading states during normal operation

One codebase, three role-scoped views

The same React application renders a different dashboard depending on the authenticated role. Agents see their own ticket queue; supervisors see team workload and can reassign; admins get configuration and audit access. Role gates run on both the router and server functions — UI-only guards aren't enough.

  • › TanStack Router file-based routes with per-role loader guards
  • › Better Auth session carries role claims, verified on every server function call
  • › Supervisors can reassign open tickets without touching the agent's local queue directly

A ticket lifecycle with a full audit trail

Tickets move through a defined set of states — Open, In Progress, Pending, Resolved, Closed — and every transition is recorded with a timestamp and the acting user. Nothing gets quietly edited; the history is always queryable.

  • › State transitions enforced server-side — an agent can't resolve a ticket assigned to someone else
  • › Each status change creates an immutable history record, not an overwrite
  • › Soft-deleted tickets preserved for reporting — a closed ticket doesn't break historical counts

Auth that survives a reload, even offline

Better Auth handles session creation and validation server-side, but the session token is also persisted in the local TanStack DB SQLite store. If the agent reloads the page while disconnected, they stay authenticated against the local session rather than being bounced to the login screen.

  • › Email/password auth via Better Auth with secure session cookies
  • › Local SQLite (OPFS) persists session data for offline continuity
  • › Session syncs to server on reconnect to validate against revocation

Offline-First

The agent keeps working. The network catches up later.

Every write — a new ticket, a status update, an agent note — lands in a local SQLite store in the browser first. The UI updates immediately from that local store. A background sync layer queues mutations and pushes them to Postgres through Prisma server functions when the connection returns.

PWA shell caching via Serwist

The application shell is cached by a Serwist service worker in production — the portal loads and is navigable even with zero network, independently of whether TanStack DB has data to show.

Background sync, not manual refresh

Architecture

One direction of trust, local-first at every write.

Writes flow down from the UI to the local SQLite store first, then up to Postgres via the sync layer. Reads come from the local store so the UI is always fast, regardless of network state.

Engineering Challenges

The decisions that were actually hard.

Offline auth without a persistent server session

Better Auth validates sessions against Postgres. When Postgres is unreachable, the agent would normally be treated as unauthenticated. The solution: persist the session token and user record in the local SQLite store, and short-circuit the auth check against the local copy when offline — with a background revalidation queued for reconnect.

Role enforcement that can't be faked client-side

Hiding UI elements is cosmetic. Every TanStack Start server function re-reads the session's role from the database and throws if the caller doesn't have the required permission — even if they manually crafted the request. The client-side role check is purely for UX; the server check is the real gate.

Ticket history without destructive updates

Rather than updating a ticket's status column and losing the previous value, every transition appends a TicketHistory row. The tradeoff is extra writes per update and a join to reconstruct current state — worth it because supervisors and auditors need to know not just what a ticket's status is, but the full sequence of how it got there.

Docker dev environment on Windows

Full project bind mounts in Docker on Windows break Vite's hot reload due to file-system notification differences between the host and container. The workaround: replace bind mounts with docker compose watch for targeted file sync, with an isolated .output volume to prevent stale SSR build artifacts from leaking across reloads.

Keeping TanStack DB and Postgres in sync during reconnect

Local writes that queue up during an offline window need to be replayed in the correct order when connectivity returns, and the server's response needs to reconcile against what the local store already shows the user. Today this uses last-write-wins, which is acceptable for a single-agent-per-session model. True multi-device conflict resolution is scoped as future work.

Type safety from database to form validation

Prisma's generated types cover the database layer. TanStack Form with Zod schemas covers the intake form. The gap between them — a form field that accepts a string but the database expects a specific enum — is closed by sharing Zod schemas between the client form and the server function validator, so a schema change propagates to both ends at once.

Tech Stack

What's in package.json, and why it's there.

React 19 + TanStack Router

File-based, fully type-safe routing. Role-guard loaders run before any component mounts, so a forbidden route never even begins to render.

TanStack Start

Full-stack React SSR with colocated server functions. No separate API server — the ticket mutation and the form that triggers it live in the same file.

TanStack DB + SQLite (OPFS)

Reactive client collections backed by an in-browser SQLite store. The local-first layer that makes offline ticket intake possible as an architectural property, not a UI workaround.

Prisma 7 + PostgreSQL

Type-safe ORM for the full schema. The pg driver adapter keeps the connection pool explicit — important when server functions can be short-lived.

Better Auth

Session-based auth with email/password. Role claims on the session token, verified server-side on every mutation. IP and user-agent logging for audit.

Serwist (PWA)

Service worker caches the application shell so the portal loads in production even with no network. Separate from TanStack DB — one caches the app, the other caches the data.

TanStack Form + Zod

Type-safe form state management paired with Zod schemas shared between the client form and the server function validator — one source of truth for what a valid ticket looks like.

Tailwind CSS v4 + shadcn/ui + Base UI

Utility-first styling with accessible Radix and Base UI primitives. Dark-mode-first — a call center portal is often running on a dim monitor all day.

Biome + Lefthook + Vitest + Playwright

Biome replaces ESLint and Prettier in one pass. Lefthook enforces it as a pre-commit gate. Vitest covers unit/integration. Playwright handles end-to-end flows.

Docker + docker compose watch

Containerized dev environment with Postgres. File sync via compose watch instead of bind mounts — the workaround that makes Vite HMR reliable on Windows.

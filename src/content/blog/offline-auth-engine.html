1. The problem — auth when there's no network

Most auth libraries are built around a single assumption: there is a server to talk to. Cookie-based sessions, JWTs, OAuth flows — all of them require a network round-trip to validate identity. That assumption breaks the moment your users work in an environment with unreliable connectivity.

NUVELCO CCA runs in field stations attached to substations across a rural province. Agents log support tickets, dispatch field crews, and update consumer accounts. When a substation loses its uplink — which happens — they need to keep working. That means they need to keep being authenticated.

The requirement is specific: an agent who has logged in at least once while online should be able to re-authenticate offline, with the same credentials, for up to seven days. After that, they must come back online.

"We can't tell a field agent their session expired and they need to connect to the internet. There is no internet."

The solution is a dual-mode credential system built on top of Better Auth. Online sessions go through the normal Better Auth flow. Offline sessions authenticate against a locally-persisted, cryptographically-hashed credential cache stored in SQLite OPFS. A single AuthEngine object coordinates both paths with the same external interface — the login page doesn't know which mode it's in until it checks navigator.onLine.

2. The architecture at a glance

Four pieces interact to make this work:

Better Auth

Handles the online session — cookies, session tokens, Prisma-backed user records. The canonical source of identity when the network is available.

localAuthCollection

A TanStack DB collection persisted in SQLite OPFS. Stores the user profile, a SHA-256 hash of the last-used password, and a 7-day expiry timestamp. The offline credential store.

AuthEngine

A plain object with four methods: loginOnline, loginOffline, syncServerToLocal, and logout. The unified interface that hides which path is active.

authStore

A TanStack Store that holds the active user identity in memory. Both auth paths write to the same store shape — the rest of the app doesn't care how the session was established.

The login form checks navigator.onLine and calls either AuthEngine.loginOnline or AuthEngine.loginOffline. Both callbacks receive a ServerUser and navigate to the same role-based landing page. Nothing downstream changes.

3. The credential cache — localAuthCollection

The cache is a TanStack DB collection backed by the same SQLite OPFS persistence layer used for all app data. It lives alongside the ticket and consumer collections but is never synced to the server — it's client-only by design. 

// src/db/local-auth.ts
import type { ServerUser } from '@/lib/better-auth/auth-server'

const LocalAuthSchema = z.object({
  id:             z.string(),
  email:          z.email(),
  hashedPassword: z.string(),  // SHA-256 hex of the last successful password
  profile:        z.object({}) as unknown as z.ZodType<LocalUser>, // full ServerUser snapshot
  expiresAt:      z.number(),   // Unix ms timestamp — Date.now() + 7 days
})

export type LocalAuth = z.infer<typeof LocalAuthSchema>

export let localAuthCollection = {} as Collection<LocalAuth, string>
if (typeof window !== 'undefined') {
  localAuthCollection = createCollection(
    persistedCollectionOptions<LocalAuth, string>({
      id: 'localAuth',
      getKey: auth => auth.id,
      persistence,      // shared SQLite OPFS instance
      schemaVersion: 1,
    }),
  )
}

A few design choices are worth calling out. The profile field stores a full snapshot of ServerUser — name, role, image, everything — so that an offline login can hydrate the auth store with the complete user object, not a partial shell. The hashedPassword is never sent to the server; it only ever lives in the browser's OPFS partition. And the collection is guarded by a typeof window !== 'undefined' check because it depends on browser APIs that don't exist during SSR.

Why OPFS and not localStorage?

localStorage is synchronous, size-limited (~5 MB), and stores everything as strings. OPFS (Origin Private File System) gives you a full SQLite database, async access, no practical size limit, and is isolated to the origin — no other site can read it. Storing a credential cache in localStorage would be careless. In OPFS it's structurally sound.

4. Online login — capturing the hash at the right moment

The online path delegates the actual authentication to Better Auth's authClient.signIn.email. The trick is what happens in the onSuccess callback: the plaintext password is hashed with crypto.subtle and written into the local cache while the credential is still available in memory. After this callback returns, the password is gone — the app never stores it. 

// src/lib/better-auth/auth-engine.ts
async loginOnline(email: string, password: string, onSuccess: (user: ServerUser) => void) {
  await authClient.signIn.email(
    { email, password },
    {
      onSuccess: async () => {
        clearSessionHotline()

        // Fetch the full user profile from the server now that the session is live
        const fullUser = await getAuthUser()
        if (!fullUser) {
          toast.error('Login succeeded but user profile could not be loaded.')
          return
        }

        // Hash the plaintext password before it leaves this scope
        const hashedPassword = await AuthEngine.hashCredentials(password)
        const localUser = [...localAuthCollection.values()].find(u => u.email === email)

        if (localUser) {
          // Update the existing entry — refresh profile snapshot and extend TTL
          await localAuthCollection.update(localUser.id, draft => {
            draft.profile        = fullUser
            draft.hashedPassword = hashedPassword
            draft.expiresAt      = Date.now() + 7 * 24 * 60 * 60 * 1000
          })
        } else {
          // First login on this device — create the entry
          await localAuthCollection.insert({
            id: fullUser.id, email: fullUser.email,
            hashedPassword,  profile: fullUser,
            expiresAt: Date.now() + 7 * 24 * 60 * 60 * 1000,
          })
        }

        onSuccess(fullUser)
      },
      onError: ctx => toast.error(ctx.error.message || 'Authentication failed'),
    },
  )
},

The hash is computed with the Web Crypto API — no library, no dependency: 

hashCredentials: async (password: string) => {
  const msgBuffer  = new TextEncoder().encode(password)
  const hashBuffer = await crypto.subtle.digest('SHA-256', msgBuffer)
  return Array.from(new Uint8Array(hashBuffer))
    .map(b => b.toString(16).padStart(2, '0'))
    .join('')
}

Every online login refreshes the cache: the profile snapshot is updated with the latest server state, the hash is refreshed with whatever password was just used, and the TTL is extended by another seven days from now. The cache is always as fresh as the most recent successful login.

5. Offline login — hash comparison and TTL enforcement

Offline login is a three-step gate: find the cached entry by email, hash the submitted password and compare it to the stored hash, check the TTL. Pass all three and the authStore is populated from the local profile snapshot. Fail any one and the attempt is rejected with an explicit error. 

async loginOffline(email: string, password: string, onSuccess: (user: ServerUser) => void) {
  // Step 1 — look up the cached entry
  const localUser = [...localAuthCollection.values()].find(u => u.email === email)
  const inputHash = await AuthEngine.hashCredentials(password)

  if (localUser && localUser.hashedPassword === inputHash) {
    // Step 2 — TTL check: don't allow stale offline sessions
    if (Date.now() > localUser.expiresAt) {
      toast.error('Offline session expired. Please connect to the internet.')
      return
    }

    clearSessionHotline()

    // Step 3 — hydrate the in-memory auth store from the local profile snapshot
    authStore.setState(s => ({
      ...s,
      user: { ...localUser.profile, assignedHotline: undefined },
      isAuthenticated: true,
    }))

    onSuccess(localUser.profile)
    return
  }

  toast.error('Invalid credentials or user not cached for offline use.')
},

A few things this deliberately does not do. It doesn't hit the network. It doesn't create a Better Auth session — there is no session token; the auth store is the session for the duration of the offline period. And it clears the assignedHotline from the restored profile: hotline assignment is a session-scoped concept (stored in sessionStorage, not in the OPFS cache) so offline users must re-select their hotline after logging in.

Why not store the hash on the server and verify there?

Because that requires a network round-trip — which is exactly what we're trying to avoid. The point of offline auth is to be fully self-contained. The hash comparison happens entirely in the browser, against data that was written to OPFS the last time the user was online. No server contact at any point.

6. Background sync — keeping the cache fresh

A credential cache that only updates on explicit login would go stale for users who stay logged in between shifts. The private route guard addresses this: every time a user navigates to a protected route while online, it calls AuthEngine.syncServerToLocal to silently refresh the cache with the latest server profile, extending the TTL. 

// Called on every protected route mount when isOnline and user exists in localAuthCollection
async syncServerToLocal(serverUser: ServerUser): Promise<void> {
  const expiry = Date.now() + 7 * 24 * 60 * 60 * 1000
  const exists = localAuthCollection.has(serverUser.id)

  if (exists) {
    await localAuthCollection.update(serverUser.id, draft => {
      draft.profile   = serverUser  // pull latest name, role, image from server
      draft.expiresAt = expiry      // extend TTL on every active session
    })
  } else {
    // User navigated to a protected route but was never explicitly logged in
    // on this device — seed the entry without a password hash
    await localAuthCollection.insert({
      id: serverUser.id, email: serverUser.email,
      profile: serverUser, hashedPassword: '',
      expiresAt: expiry,
    })
  }
}

Note the hashedPassword: '' in the insert branch. An entry seeded this way will never pass offline auth — the empty hash will never match a real password hash. This is intentional: a user who has never explicitly logged in on this device doesn't have a cached password. Seeding the profile without a hash ensures the cache is populated for the next online login to update, rather than silently failing or being absent entirely.

This also means syncServerToLocal is safe to call on every route mount. It only updates the profile and TTL — it never touches hashedPassword unless the entry already has one.

7. The auth store — hotline state and idempotent setUser

The auth store is a TanStack Store — a lightweight reactive container with no React dependency. It holds the active ServerUser in memory and a handful of boolean flags. Both the online and offline auth paths write to the same store, so every consumer in the component tree sees the same shape regardless of how the session was established. 

// src/store/auth-store.ts
export const authStore = new Store<AuthState>({
  isAuthenticated: false,
  isLoggingOut:    false,
  user:            {} as unknown as ServerUser,
  // ...other flags
})

Two subtle patterns in the store are worth explaining. First, the setUser function is idempotent — if the store already has a user, it returns the existing state untouched. This prevents the server-side beforeLoad from overwriting a user that was already restored from the offline cache on a back-navigation: 

export const setUser = (user: ServerUser) => {
  authStore.setState(state => {
    if (state.user.id) return state  // already have a user — don't overwrite

    return user
      ? { ...state, isAuthenticated: true, user: withSessionHotline(user) }
      : defaultValue
  })
}

Second, the hotline is deliberately not stored in OPFS or in the Better Auth session model directly. It lives in sessionStorage — scoped to the browser tab, cleared on close. When setUser runs, it reads the hotline from sessionStorage and merges it into the user object before storing it. This means the hotline survives a page reload within the same tab but is gone after a fresh login, which matches the intended UX: agents pick their hotline at the start of each shift. 

function withSessionHotline(user: ServerUser): ServerUser {
  const assignedHotline = readSessionHotline()  // reads from sessionStorage
  return assignedHotline
    ? { ...user, assignedHotline }
    : { ...user, assignedHotline: undefined }
}

8. The private route guard — online vs. offline identity resolution

The private layout route is where the two auth paths converge. It has to answer one question: is there a valid authenticated user right now? The answer depends on whether the app is online or offline. 

// src/routes/(private)/route.tsx
function RouteComponent() {
  const isOnline       = useIsOnline()
  const { user }       = Route.useRouteContext()  // from server beforeLoad
  const localAuths     = useLiveQuery(q => q.from({ localAuth: localAuthCollection }))

  useEffect(() => {
    if (!localAuths.isReady) return

    // Online: identity comes from the server beforeLoad context
    // Offline: identity comes from the in-memory auth store (set at offline login)
    const localUser = isOnline ? user : authStore.state.user

    if (!localUser?.id) {
      navigate({ to: '/login' })
      return
    }

    // If online and this user exists in the local cache, refresh the cache entry
    const exists = localAuths.data.find(u => u.id === localUser.id)
    if (exists && isOnline && user) {
      AuthEngine.syncServerToLocal(user)
    }
  }, [localAuths.isReady, navigate, user])

  if (!user) return <Loading />
  return <Outlet />
}

When online, user comes from the root's beforeLoad function which calls getAuthUser() — a server function that validates the Better Auth cookie and fetches the user from Postgres. When offline, that server call fails silently (the beforeLoad catches the error and returns { user: undefined }), so the guard falls back to authStore.state.user — which was set by loginOffline.

The useLiveQuery on localAuthCollection isn't used for the auth check itself — it's used to gate the syncServerToLocal call. The collection needs to be ready before we can check if the user exists in it. Once it is, the sync runs exactly once per route mount while online.

9. The verifyAuth trick — checking credentials without clobbering the session

There's a second, separate auth scenario in the app: an admin needs to re-confirm a logged-in agent's identity before a destructive action, without that check ending the agent's current session. The naive approach — calling authClient.signIn.email — would create a new session and overwrite the active session token. That's a silent bug that's easy to miss.

The solution is verifyAuth, a server function that calls Better Auth's internal auth.api.signInEmail with asResponse: true. This flag tells Better Auth to run the credential check and return a plain HTTP Response instead of its normal behavior of writing session cookies. The active session is completely untouched. 

// src/lib/better-auth/auth-server.ts
export const verifyAuth = createServerFn({ method: 'POST' })
  .middleware([authMiddleware])
  .handler(async ({ data }) => {
    const response = await auth.api.signInEmail({
      body:  { email: data.email, password: data.password },
      asResponse: true,  // ← the key: returns a Response, sets no cookies
    })

    if (!response.ok) {
      return { success: false, error: 'Invalid password.' }
    }

    // Pull minimal user data — just enough for the caller to act on
    const user = await prisma.user.findFirst({
      where: { email: data.email },
      select: { id: true, firstName: true, lastName: true, role: true },
    })

    return user
      ? { success: true, data: user }
      : { success: false, error: 'User is not authorized.' }
  })

The distinction between authClient.signIn.email (client-side, writes cookies, creates a session) and auth.api.signInEmail with asResponse: true (server-side, stateless check, no side effects) is easy to conflate. Missing it would create a subtle multi-user bug that only shows up in production when two people are logged in on the same machine.

asResponse: true is not documented prominently in Better Auth's guides. It's available on all auth.api.* methods and is the correct pattern any time you need to use the auth logic as a pure function without session side effects.

10. Logout — cleaning up both worlds

Logout needs to clean up both the server session and the in-memory state. The implementation is deliberately optimistic: the UI is cleared immediately regardless of whether the server signout succeeds, and the server call only fires if the app is online. An offline logout clears local state and redirects without touching the server. 

async logout(params: { onSuccess: () => void }) {
  // Immediately mark as logging out — blocks UI interactions
  authStore.setState(state => ({ ...state, isLoggingOut: true }))

  // Clear ephemeral state in both overlay and sessionStorage
  clearModals()
  clearSessionHotline()
  resetAuth()  // returns authStore to defaultValue synchronously

  try {
    if (navigator.onLine) {
      // Best-effort server signout — invalidates the session token
      await authClient.signOut({}, params)
    }
  } catch (error) {
    console.error('AuthEngine: Server signOut failed', error)
  }

  params.onSuccess()  // navigate to /login in all cases
}

The local credential cache (localAuthCollection) is intentionally not cleared on logout. The rationale: the cache exists for connectivity resilience. Clearing it on logout would mean an agent who logs out before going offline can't log back in. The cache entry expires naturally via TTL, and a future online login will refresh it. If you need a "clear this device" capability, that's a separate explicit action.

12. Tradeoffs and what this doesn't solve

What this gets right

  • › Zero network dependency on the offline path — no background sync, no service worker proxy, no queued requests
  • › The hash never leaves the browser. The plaintext password is available in memory for exactly the duration of the onSuccess callback, then gone
  • › Both auth paths produce the same ServerUser shape — the rest of the app is unaware which path was used
  • › TTL enforcement means offline access has a hard expiry that can't be extended without coming back online
  • › syncServerToLocal runs on every protected route mount — active users stay fresh without any explicit refresh action

What it doesn't solve

  • › No server-side revocation. If an account is disabled on the server while the device is offline, the local cache has no way to know. The user can log in offline until their TTL expires
  • › SHA-256 is not a password hash. It's fast, which makes it unsuitable for protecting credentials against a determined attacker with access to the OPFS database. For a higher-security requirement, this should use PBKDF2 or bcrypt via a WASM port
  • › No multi-device invalidation. Clearing the credential cache on a lost or reassigned device requires a separate admin action — there's no remote wipe mechanism
  • › First login always requires connectivity. An agent who has never logged in on a given device can't do so offline — the hash only exists after a successful online login

For NUVELCO CCA — a known-device, known-user, internally-operated system where accounts are managed by an admin — these tradeoffs are acceptable. SHA-256 isn't protecting against a cloud breach; it's protecting against someone picking up an unattended tablet and guessing a password. The TTL and the absence of a network path mean the blast radius of a compromised device is bounded by time and scope.

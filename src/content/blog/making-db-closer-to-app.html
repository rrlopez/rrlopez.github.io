1. The problem — the gap between DB and app

Most full-stack apps have an invisible wall between the database and the client. On one side you have a Prisma schema with precise model types. On the other you have a React component expecting Ticket[]. Between them there's a REST route that returns any, a fetch call that returns unknown, and a chain of manual casts hoping nothing changed since last deploy.

The architecture in this post comes from NUVELCO CCA — an offline-first customer call and operations management system built for a regional utility provider. Agents log tickets, dispatch field crews, and track consumer accounts from field stations that routinely lose connectivity, so the app needed a local data layer that could operate completely disconnected and reconcile with the primary Postgres database when a connection was restored.

The goal was to close that gap completely — no manually maintained interface files that duplicate the schema, no as SomeType casts, no silent runtime mismatches. When the Prisma schema changes, TypeScript should tell you every place in the app that needs to catch up.

"I want the IDE to autocomplete ticket.concernCategory in a component the same way it does inside a Prisma query."

That required designing a clear five-layer pipeline: schema → CRUD API → Transaction API → Collection factory → UI. Each layer is thin, and the types flow through without any layer introducing an any break.

2. Layer 1 — Prisma schema as the single source of truth

Everything starts at prisma/schema.prisma. The schema defines the domain — Ticket, Consumer, Activity, enums like TicketStatus and TicketPriority — and Prisma's generator emits TypeScript types from it. No interface file is ever written by hand. 

// prisma/schema.prisma (excerpt)
enum TicketStatus {
  PENDING
  IN_PROGRESS
  RESOLVED
  CANCELED
}

model Ticket {
  id               String         @id @default(uuid())
  channel          TicketChannel  @default(CALL)
  hotline          String
  concernCategory  String
  concernCause     String?
  status           TicketStatus   @default(PENDING)
  priority         TicketPriority @default(MEDIUM)
  agentId          String
  consumerId       String?
  reportDate       DateTime       @default(now())
  // ... other fields
  @@map("tickets")
}

Prisma generates two outputs. The server-side client lives at prisma/generated/prisma/client.ts and is only imported in server code. The frontend gets a separate, safe export: 

// prisma/generated/prisma/browser.ts  (auto-generated, do not edit)
export type Ticket         = Prisma.TicketModel
export type Consumer       = Prisma.ConsumerModel
export type Activity       = Prisma.ActivityModel
// ... all 11 models
export * from './enums'  // TicketStatus, TicketPriority, Role, etc.

This split matters. The frontend imports from prisma/generated/prisma/browser — never from the client module, which would bundle Postgres adapters and server-only internals into the client bundle. One generated file, two safe entry points.

3. Layer 2 — The CRUD API: a type-safe Proxy server function

The first time you see crudAPI.ticket('findMany', { where: { status: 'PENDING' } }) from a component, it looks like you're calling Prisma directly from the browser. That's the point. Under the hood it's a TanStack server function, but the interface mirrors Prisma's delegate API exactly — including the return type.

The trick is four TypeScript utility types that work together to make the Proxy fully inferred: 

// src/lib/prisma-client/crud-api.ts
type DB             = typeof prisma
type ModelName      = Uncapitalize<Prisma.ModelName>
type DelegateMethods = 'findMany' | 'findFirst' | 'findUnique'
                    | 'create' | 'update' | 'updateMany'
                    | 'upsert' | 'delete' | 'deleteMany'
                    | 'count' | 'groupBy'

// Flattens intersections to a single readable object type
type DeepPrettify<T> = T extends Date
  ? T
  : T extends object
    ? { [K in keyof T]: DeepPrettify<T[K]> } & {}
    : T

// The exact args shape that Prisma expects for a given model + method
type CleanArgs<T extends ModelName, M extends DelegateMethods>
  = Parameters<DB[T][M]>[0]

// The exact return type that Prisma would give back
type InferResult<T extends ModelName, M extends DelegateMethods, A>
  = Prisma.Result<DB[T], A, M>

Those four types snap together into a single CrudProxy mapped type that covers every model and every method: 

type CrudProxy = {
  [K in ModelName]: <M extends DelegateMethods, A extends CleanArgs<K, M>>(
    action: M,
    args?: A
  ) => Promise<Result<DeepPrettify<InferResult<K, M, A>>, string>>
}

The runtime side is a JavaScript Proxy that intercepts property access, packages the table name, action, and args into a plain DBPayload, and fires a TanStack server function with an auth middleware guard: 

// The shared execution engine — used by both CRUD and Transaction APIs
export async function executeOperation(dbInstance: any, payload: DBPayload) {
  const delegate = dbInstance[payload.table]
  if (!delegate?.[payload.action]) {
    throw new Error(`Invalid operation: ${payload.action} on ${payload.table}`)
  }
  return await delegate[payload.action](payload.args)
}

// Server function — runs on the server only
const crudServerFn = createServerFn({ method: 'POST' })
  .middleware([authMiddleware])
  .handler(async ({ data }) => {
    const result = await ResultAsync.fromPromise(
      executeOperation(prisma, data),
      (e: any) => e.message || 'Database operation failed'
    )
    return result.isOk() ? { value: result.value } : { error: result.error }
  })

// Public API — the Proxy that makes it feel like calling Prisma
export const crudAPI = new Proxy({} as CrudProxy, {
  get(_, table: string) {
    return async (action: string, args: any) => {
      const response = await crudServerFn({ data: { table, action, args } })
      if ('error' in response) return err(response.error as string)
      return ok(response.value as any)
    }
  },
})

The result is always a neverthrow Result type — no try/catch, no unhandled promise rejections. Callers always handle both .isOk() and .isErr() branches explicitly.

Why Prisma.Result<> is the key

Prisma ships a conditional type called Prisma.Result<Model, Args, Operation> that computes the return type for any model + method + args combination — including select and include shapes. By threading A (the inferred args generic) through InferResult, the proxy's return type automatically narrows based on whatever args you pass. Pass { select: { id: true, status: true } } and you get back { id: string; status: TicketStatus } — not a full Ticket.

4. Layer 3 — The Transaction API: atomic multi-table writes

Creating a ticket isn't one write. It might involve inserting a Contact, upserting a Consumer, linking a ConsumerAddress, and inserting the Ticket itself — all of which need to either all succeed or all roll back together. The CRUD API handles one operation at a time. The Transaction API handles a batch.

The key design decision was to reuse executeOperation — the same function that powers the CRUD API — but pass a Prisma tx client instead of the global prisma instance: 

// src/lib/prisma-client/transaction-api.ts
const transactionServerFn = createServerFn({ method: 'POST' })
  .middleware([authMiddleware])
  .handler(async ({ data }) => {
    const txResult = await ResultAsync.fromPromise(
      prisma.$transaction(async tx => {
        const executionResults = []
        for (const op of data.operations) {
          // Same engine, different client — tx rolls everything back on failure
          const out = await executeOperation(tx, op)
          executionResults.push(out)
        }
        return executionResults
      }),
      (e: any) => e.message || 'Transaction batch execution failed'
    )
    return txResult.isOk()
      ? { value: txResult.value }
      : { error: txResult.error }
  })

export const transactionAPI = {
  execute: async (operations: DBPayload[]): Promise<Result<any[], string>> => {
    const response = await transactionServerFn({ data: { operations } })
    if ('error' in response) return err(response.error as string)
    return ok(response.value)
  },
}

The input is DBPayload[] — the exact same shape the CRUD API uses internally. That means the client assembles a list of plain objects describing what to do, sends them over in one round-trip, and Prisma wraps the whole batch in a real database transaction. If any operation throws, nothing is committed.

What this gives you

  • › One HTTP round-trip for multi-table writes
  • › True DB-level atomicity via prisma.$transaction()
  • › No duplicated validation logic — same executeOperation engine
  • › Server results returned in index order — each operation maps back to its collection

What it doesn't do

  • › It's not a query API — reads go through the CRUD Proxy individually
  • › No partial rollback — all-or-nothing per batch
  • › Return types are any[] — the caller knows the shape by position

5. Layer 4 — Collection factory: bringing the DB to the client

The CRUD API bridges the server. But components don't call crudAPI directly — they read from local collections. A collection is an in-memory, SQLite-persisted (via OPFS), reactive store that stays in sync with the server. Think of it as a lightweight local replica of a Prisma table.

It's worth being precise about how that replication actually works, because it's deliberately simple. There is no long-lived WebSocket or CDC stream. Synchronization happens through the same CRUD and Transaction server functions already described — the collection's queryFn calls crudAPI.findMany to pull a fresh snapshot from Postgres on demand, and every write path goes through transactionAPI.execute, which writes to Postgres first and then patches the local SQLite store with the server-confirmed rows. The OPFS layer is purely a persistence cache: it lets the app boot instantly from the last-known state and serve reads while offline, but the server is always the source of truth. Cross-tab consistency is handled by a BroadcastChannel that signals other open tabs to re-sync when any tab commits a write. If you needed continuous real-time replication — streaming CDC from Postgres to the edge — you would slot in a tool like ElectricSQL or PowerSync at the queryFn boundary without touching any other layer. For NUVELCO CCA's workload — periodic syncs on connectivity restore, not sub-second live updates — pull-on-demand is the right tradeoff.

The factory function createSyncableCollection<TRecord> takes a Prisma browser model type as its generic and wires up everything — persistence, query sync, optimistic inserts/updates/deletes, offline detection, and cross-tab sync via BroadcastChannel: 

// src/db/index.tsx (simplified)
export function createSyncableCollection<TRecord extends { id: string }>(
  config: CreateSyncableCollectionConfigs
): Collection<TRecord> {
  const collection = createCollection(
    persistedCollectionOptions({
      id: config.apiKey,
      persistence,          // SQLite OPFS — survives page reload
      schemaVersion: config.schemaVersion,
      ...queryCollectionOptions({
        queryKey: [config.apiKey],
        syncMode: config.syncMode,   // 'eager' or 'on-demand'
        getKey: (item: TRecord) => item.id,

        // Fetch from server via the CRUD Proxy
        queryFn: async ctx => {
          if (!navigator.onLine) return []
          const result = await api[config.apiKey]('findMany', { where, orderBy, take })
          if (result.isErr()) throw new Error(result.error)
          return result.value
        },

        // Optimistic insert — writes locally, then confirms with server
        onInsert: async ({ transaction }) => {
          const results: TRecord[] = []
          for (const mutation of transaction.mutations) {
            if (!navigator.onLine) { results.push(mutation.modified); continue }
            const value = await retryWithBackoff(() =>
              api[config.apiKey]('create', { data: mutation.modified })
            )
            results.push(value as TRecord)
          }
          return results
        },

        // onUpdate and onDelete follow the same pattern...
      }),
    })
  )

  // Cross-tab sync: broadcast on change, re-sync on receive
  bc.addEventListener('message', async event => {
    if (event.data.apiKey !== config.apiKey) return
    if (event.data.fromTab === dbSyncTabId) return  // ignore own broadcasts
    await collection._sync.startSync()
  })
  collection.subscribeChanges(() => {
    bc.postMessage({ apiKey: config.apiKey, fromTab: dbSyncTabId })
  })

  return collection
}

Creating a collection for each Prisma model is then just a one-liner. All eleven models live in src/db/collections.ts: 

// src/db/collections.ts
import type { Ticket, Consumer, Activity /* ... */ }
  from 'prisma/generated/prisma/browser'
import { createSyncableCollection } from '.'

const SCHEMA_VERSION = 14

export const ticketCollection   = createSyncableCollection<Ticket>({
  apiKey: 'ticket', schemaVersion: SCHEMA_VERSION, syncMode: 'on-demand',
})
export const consumerCollection = createSyncableCollection<Consumer>({
  apiKey: 'consumer', schemaVersion: SCHEMA_VERSION, syncMode: 'on-demand',
})
export const activityCollection = createSyncableCollection<Activity>({
  apiKey: 'activity', schemaVersion: SCHEMA_VERSION, syncMode: 'on-demand',
})
// ... 8 more collections, all typed to their Prisma model

The syncMode flag lets some collections load eagerly on mount (user, address, operationsTeam — small, shared reference data) while others load on-demand only when a component actually subscribes. The SQLite OPFS layer means the last-fetched data is available instantly even after a hard reload, before any server sync completes.

6. Layer 5 — dbTransaction(): the local write orchestrator

Individual collection inserts and updates work for simple cases. But creating a ticket touches six or seven collections at once and all of those writes need to be atomic — either they all land in the database or none of them do. That's what dbTransaction() is for.

It wraps TanStack DB's createTransaction, which collects all collection mutations made inside the callback into a single Transaction object. The mutationFn then serializes those mutations into DBPayload[] and fires the Transaction API in one shot: 

// src/db/local-db-transaction.ts
function buildOperations(transaction: Transaction<any>): DBPayload[] {
  return transaction.mutations.map(mutation => {
    if (mutation.type === 'insert')
      return { table: mutation.collection.id, action: 'create', args: { data: mutation.modified } }
    if (mutation.type === 'update')
      return { table: mutation.collection.id, action: 'update',
                args: { where: { id: mutation.original.id }, data: mutation.changes } }
    return   { table: mutation.collection.id, action: 'delete',
                args: { where: { id: mutation.original.id } } }
  })
}

export const dbTransaction = (callback: () => void): ResultAsync<unknown[], Error> => {
  return ResultAsync.fromPromise((async () => {
    const isOffline = !navigator.onLine
    let serverResults: unknown[] = []

    const tx = createTransaction({
      mutationFn: async ({ transaction }) => {
        if (isOffline) {
          // Offline: apply mutations locally, queue for later sync
          serverResults = await applyLocalTransaction(transaction)
          return serverResults
        }

        // Online: send all ops in one DB transaction
        const result = await transactionAPI.execute(buildOperations(transaction))
        if (result.isErr()) throw new Error(result.error)

        // Apply server-returned rows directly to each local collection
        await syncCollectionsAfterManualTransaction(transaction, result.value)
        serverResults = result.value
        return result.value
      },
    })

    tx.mutate(callback)
    await tx.isPersisted.promise
    return serverResults
  })(), error => error instanceof Error ? error : new Error(String(error)))
}

After the server responds, syncCollectionsAfterManualTransaction patches each collection directly with the server-returned rows — no extra findMany round-trip. If a targeted write fails, it falls back to a full collection refetch. Either way, the local store reflects exactly what landed in the database.

The callback passed to dbTransaction() is just plain collection calls — ticketCollection.insert(…), consumerCollection.update(…). The framework intercepts them, batches them, and handles the rest.

7. The UI layer — forms, live queries, and the full picture

The outermost layer is the one users interact with. A form validated by Zod, state read via useLiveQuery, and writes dispatched through dbTransaction(). No direct fetch, no manual state management, no useEffect data loading.

The Zod schema in -ticket-form-view.tsx defines the form shape and TanStack Form validates against it on change. The inferred type TicketIntakeFormData drives handleCreate's parameter type — which means if you rename a field in the schema, the compiler will complain in the handler before anything ever runs: 

// src/routes/(private)/(dashboard)/tickets/create/-ticket-form-view.tsx
export const ticketIntakeSchema = z.object({
  reportDate:           z.string().min(1),
  channel:              z.enum(TICKET_CHANNELS),
  concernLeafChoice:    z.string().min(1),
  status:               z.enum(['PENDING', 'IN_PROGRESS', 'RESOLVED', 'CANCELED']),
  priority:             z.enum(['LOW', 'MEDIUM', 'HIGH', 'CRITICAL']),
  barangayMunicipality: z.string().min(1),
  // ...other fields
})

export type TicketIntakeFormData = z.infer<typeof ticketIntakeSchema>

Reading data in the component is a useLiveQuery call — reactive, live-updating, typed to the collection's generic parameter: 

// src/routes/(private)/(dashboard)/tickets/create/-create-ticket-from.tsx
const { data: consumers = [] } = useLiveQuery(
  q => q.from({ consumers: consumerCollection })
)
// consumers: Consumer[]  — typed from Prisma's generated Consumer model

And writing is just calling dbTransaction() with a callback that orchestrates all collection mutations in plain imperative code: 

const handleCreate = async (value: TicketIntakeFormData) => {
  const result = await dbTransaction(() => {
    const ticketId = crypto.randomUUID()

    // Insert or resolve contact
    let contactId = contacts.find(c => c.number === value.phoneNumber)?.id
    if (!contactId && value.phoneNumber) {
      contactId = crypto.randomUUID()
      contactCollection.insert({ id: contactId, number: value.phoneNumber, createdAt: new Date() })
    }

    // Upsert consumer
    const existingConsumer = consumers.find(c => c.accountNumber === value.accountNumber)
    let consumerId: string
    if (existingConsumer) {
      consumerId = existingConsumer.id
      consumerCollection.update(consumerId, draft => {
        if (value.firstName) draft.firstName = value.firstName
        draft.updatedAt = new Date()
      })
    } else {
      consumerId = crypto.randomUUID()
      consumerCollection.insert({ id: consumerId, /* ...fields */ })
    }

    // Insert the ticket — typed to the Ticket model via ticketCollection's generic
    ticketCollection.insert({
      id: ticketId,
      agentId: user.id,
      consumerId,
      contactId: contactId || null,
      status: value.status,
      priority: value.priority,
      concernCategory: category,
      hotline: user.assignedHotline,
      // ...all fields fully typed — the compiler will catch any mismatch
    })
  })

  if (result.isErr()) toast.error(result.error.message)
}

Notice there's no explicit type annotation on the insert call — TypeScript already knows the shape because ticketCollection is Collection<Ticket>. Pass the wrong field name or the wrong value for status and the IDE flags it immediately, before any server round-trip.

8. Bonus: the soft-delete extension

One more piece worth calling out. Some models — User and Activity — have a deletedAt field. Rather than rely on every developer to remember to filter soft-deleted records and to never call .delete(), the Prisma client itself enforces the policy via an extension: 

// src/lib/prisma-client/soft-delete-extension.ts
export const softDeleteExtension = Prisma.defineExtension({
  name: 'softDelete',
  query: {
    $allModels: {
      async $allOperations({ model, operation, args, query }) {
        const hasDeletedAtField = 'deletedAt' in Prisma[`${model}ScalarFieldEnum`]

        // If the model doesn't have deletedAt, pass through untouched
        if (!hasDeletedAtField) return query(args)

        // Hard deletes are forbidden — must use .update({ deletedAt: new Date() })
        if (operation === 'delete' || operation === 'deleteMany') {
          throw new Error(
            `[Prisma Error]: Hard deletes are FORBIDDEN on model "${model}". ` +
            `Use .update() to set "deletedAt: new Date()" instead.`
          )
        }

        // Reads automatically exclude soft-deleted rows
        const readOps = ['findMany', 'findFirst', 'findUnique', 'count']
        if (readOps.includes(operation)) {
          args.where = { ...args.where, deletedAt: null }
        }

        return query(args)
      },
    },
  },
})

The extension is applied once at client instantiation using ReturnType to keep the complex extension types intact: 

// src/lib/prisma-client/index.ts
const createBaseClient = () =>
  new PrismaClient({ adapter }).$extends(softDeleteExtension)

// ReturnType preserves the full extended type — don't use PrismaClient directly here
type BasePrismaClient = ReturnType<typeof createBaseClient>

const globalForPrisma = global as unknown as { prisma: BasePrismaClient }
export const prisma = globalForPrisma.prisma || createBaseClient()

From this point, every read operation anywhere in the app — whether called directly, through the CRUD Proxy, or through the Transaction API — is guaranteed to exclude soft-deleted records. The policy is structural, not documented.

10. Key takeaways

One source of truth

The Prisma schema is the only place types are defined by hand. Every layer — server functions, collections, form schemas, components — derives its types from the generated output. Add a field to the schema, run prisma generate, and TypeScript tells you everywhere that needs to update.

One execution engine

executeOperation() is a twelve-line function. Both the CRUD API and Transaction API use it. The CRUD path passes the global prisma instance; the transaction path passes a tx client. Validation, error handling, and formatting are never duplicated.

Policy is structural, not documented

The soft-delete extension makes it impossible to hard-delete a protected model or forget to filter deleted records on a read. Auth is enforced at the server function boundary via middleware — not scattered across individual handlers. These constraints cannot be accidentally bypassed.

Offline is a first-class concern

SQLite OPFS persistence means the app loads with data from last session before the first server sync. Every write path has an explicit navigator.onLine check — offline mutations are applied locally and queued. The UI never blocks on connectivity.

Components stay clean

Components don't import crudAPI, don't call fetch, don't manage loading state for mutations. They read from collections with useLiveQuery and write with dbTransaction(). The infrastructure is invisible until something goes wrong — and neverthrow's Result type ensures errors are always handled explicitly.

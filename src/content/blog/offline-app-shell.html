Service workers, OPFS headers, SSR hydration gates, and a connectivity probe that doesn't lie

1. Context — what "offline" actually means here

NUVELCO CCA is an offline-first operations management system for a regional utility provider. Field agents log service tickets, manage consumer records, and dispatch field crews from substations that lose their uplink without warning. "Offline support" isn't a nice-to-have — it's the baseline.

The previous posts in this series covered the data layer (Prisma → CRUD API → TanStack DB collections with SQLite OPFS persistence) and the auth layer (dual-mode credential system with a locally-cached SHA-256 hash). This post covers the layer below both of those: the app shell — the infrastructure that makes the app loadable, navigable, and functional with no network at all.

Specifically: how the service worker is built and configured, why the server function routes are excluded from every cache strategy, what cross-origin isolation headers are and why OPFS won't work without them, how the SSR boundary is managed to stop browser-only code from running on the server, and how a simple connectivity probe avoids lying to the rest of the app about the network state.

2. The five layers of the shell

01

Cross-origin isolation headers

Required by the browser before it will grant access to SharedArrayBuffer and OPFS. Set at the Vite dev server and Nitro preview server level.

02

Custom Vite plugin

Builds the service worker at the right moment — during buildStart for dev (so the file exists immediately), during closeBundle for production (after hashed assets are known, so the precache manifest is accurate).

03

Service worker (sw.ts)

A Serwist-powered SW with a hard prod/dev split. In dev: NetworkOnly for everything. In prod: NetworkFirst for navigation, CacheFirst for static assets, and NetworkOnly for /_serverFn — always.

04

useIsOnline

A hook that doesn't trust navigator.onLine. It performs a real HEAD request to /favicon.ico with a cache-busting timestamp to confirm actual connectivity.

05

AppWrapper

A thin component that gates all browser-only code behind a client-side hydration check. Nothing that touches navigator, OPFS, or sessionStorage renders on the server.

Each layer is independent. You could swap out Serwist for a hand-rolled service worker, or replace the Vite plugin with a separate build step, without touching anything else. The contracts between layers are narrow: headers are an HTTP concern, the SW is a separate process, the online probe is a hook, the hydration gate is a React component.

3. Cross-origin isolation — the prerequisite for OPFS

The SQLite OPFS persistence layer used by TanStack DB requires SharedArrayBuffer, which browsers only allow in cross-origin isolated contexts. Without the right HTTP headers, the SQLite WASM module will throw and no offline storage will work — silently, depending on the error boundary setup.

Three headers are required, set on every response from both the dev server and the production preview server: 

// vite.config.ts — applied to dev server and preview
const crossOriginIsolationHeaders = {
  'Cross-Origin-Opener-Policy':    'same-origin',
  'Cross-Origin-Embedder-Policy':  'require-corp',
  'Cross-Origin-Resource-Policy': 'cross-origin',
}

export default defineConfig({
  server:  { ...serverConfig, headers: crossOriginIsolationHeaders },
  preview: { headers: crossOriginIsolationHeaders, host: true, port },
})

What each header does:

  • ›
    Cross-Origin-Opener-Policy: same-origin — isolates the browsing context from cross-origin popups. Required for SharedArrayBuffer re-enablement after Spectre mitigations.
  • ›
    Cross-Origin-Embedder-Policy: require-corp — requires all sub-resources to either be same-origin or explicitly opt in with a Cross-Origin-Resource-Policy header. This is the policy that actually gates SharedArrayBuffer.
  • ›
    Cross-Origin-Resource-Policy: cross-origin — allows this origin's resources to be loaded by other origins. Prevents the embedder policy above from blocking legitimate cross-origin assets like fonts or CDN scripts.

Production note

These headers must also be set by your production host (Vercel, Cloudflare, Nginx, etc.) — Vite's config only covers dev and the local preview server. If you deploy to Vercel, add them to vercel.json under headers. Missing them in production means OPFS silently fails to initialise and every collection falls back to an empty in-memory state on load.

4. The custom Vite plugin — building the SW at the right time

Standard Serwist/Workbox Vite plugins are designed for SPA build pipelines. TanStack Start uses Nitro as its server, which emits output to .output/public rather than dist. The plugin ecosystem doesn't handle this out of the box, so the project ships a custom Vite plugin: tanstackSerwistPlugin.

The plugin hooks into two Vite lifecycle events with different responsibilities: 

// vite-plugin.ts
export function tanstackSerwistPlugin(): Plugin {
  return {
    name: 'tanstack-serwist',

    configResolved(config) {
      rootDir      = config.root
      isProduction = config.command === 'build'
    },

    async buildStart() {
      // DEV: build the SW immediately so the browser has a /sw.js to register
      // Output goes to /public — served by Vite's dev server
      if (!isProduction) await buildServiceWorker(rootDir, false)
    },

    async closeBundle() {
      // PROD: wait until the full app bundle is complete
      // Only then do we know the hashed asset filenames for the precache manifest
      // Output goes to .output/public — Nitro's static asset directory
      if (isProduction) await buildServiceWorker(rootDir, true)
    },
  }
}

The buildServiceWorker function runs in two steps for production: first it bundles src/sw.ts as an ES module library with Vite, then it calls injectManifest from @serwist/build to replace the self.__SW_MANIFEST placeholder with the actual list of hashed asset URLs: 

async function buildServiceWorker(rootDir: string, production: boolean) {
  const outDir  = production
    ? path.resolve(rootDir, '.output', 'public')  // Nitro's output dir
    : path.resolve(rootDir, 'public')              // Vite's dev server root

  // Step 1: bundle sw.ts → sw.js
  await build({
    configFile: false,
    define: { 'process.env.NODE_ENV': JSON.stringify(production ? 'production' : 'development') },
    build: {
      lib:     { entry: swSrc, formats: ['es'], fileName: () => outName },
      outDir,  emptyOutDir: false,  minify: production,
    },
  })

  // Step 2 (prod only): inject the precache manifest into the bundled sw.js
  if (production) {
    const result = await injectManifest({
      swSrc: swDest, swDest,
      globDirectory: outDir,
      globPatterns: ['**/*.{js,css,html,png,svg,ico,json,woff,woff2}'],
      injectionPoint: 'self.__SW_MANIFEST',
    })
    console.info(`✅ [SERWIST] Precached ${result.count} files`)
  }
}

The two-step approach matters: if you inject the manifest before bundling, the asset list won't include the hashed filenames Vite generates. If you bundle after the app is built but before closeBundle fires, the output directory may not exist yet. The hook timing is the entire point of writing a custom plugin rather than using an off-the-shelf one.

5. The service worker — a caching strategy that knows what it can't cache

The service worker is written in TypeScript (src/sw.ts) and compiled by the Vite plugin above. The core of the strategy is a hard environment split controlled by a single isProd flag, with a deliberate rule at the center: server functions (/_serverFn/*) are always NetworkOnly in every environment. 

const isProd = typeof process !== 'undefined'
  ? process.env['NODE_ENV'] === 'production'
  : import.meta.env?.MODE === 'production'

const serwist = new Serwist({
  precacheEntries: isProd ? (self.__SW_MANIFEST ?? []) : [],  // dev: no precache
  skipWaiting:     true,   // new SW activates immediately, no waiting
  clientsClaim:    true,   // immediately controls all open tabs
  navigationPreload: isProd, // disabled in dev to avoid 500 errors on SW startup

  runtimeCaching: isProd
    ? [
        // Navigation: try network first, fall back to cache
        // Handles hard reload when offline — serves last-cached HTML shell
        {
          matcher: ({ request }) => request.mode === 'navigate',
          handler: new NetworkFirst({ cacheName: 'pages-cache' }),
        },
        // Static assets: cache first, zero network cost on repeat visits
        {
          matcher: ({ request }) =>
            request.destination === 'style'  ||
            request.destination === 'image'  ||
            request.destination === 'font',
          handler: new CacheFirst({ cacheName: 'static-assets' }),
        },
        // Server functions: NEVER cache — always go to the network
        // Caching a server function response would be catastrophically wrong
        {
          matcher: ({ url }) => url.pathname.startsWith('/_serverFn'),
          handler: new NetworkOnly(),
        },
      ]
    : [
        // Dev: pass everything through — no caching at all
        // Eliminates the class of bugs where a stale cached response
        // is served during active development
        { matcher: () => true, handler: new NetworkOnly() },
      ],
})

Three design decisions are worth examining individually.

Why NetworkOnly for /_serverFn?

TanStack Start server functions are POST requests that hit /_serverFn. Caching a server function response would mean a stale authentication check, a stale database read, or worse — a cached write response that the server never actually executed. The rule is non-negotiable: if it talks to the server, the SW never intercepts it.

Why NetworkFirst for navigation (not CacheFirst)?

The HTML shell returned by a navigation request is server-rendered and may contain fresh data. CacheFirst would serve a stale HTML response even when online, which could show outdated beforeLoad data. NetworkFirst tries the network and only falls back to the cache when the network is genuinely unreachable — giving you offline navigation without trading away freshness when you're connected.

Why disable navigationPreload in dev?

Navigation Preload sends the navigation request to the server in parallel with SW startup, then the SW decides whether to use the preloaded response or fetch fresh. In dev the SW is rebuilt on every buildStart, which means there's a window where the SW is activating but not yet ready. Navigation Preload during that window can produce a 500. Disabling it in dev means navigations wait for the SW but that's fine — in prod it's re-enabled and the handlers are stable.

Finally, the NavigationRoute registration gates precaching to production only. In dev there are no precache entries and no navigation route — the SW is essentially a transparent pass-through that exists solely so the browser's "service worker installed" flag is set, enabling OPFS access. 

// Only register the precache navigation handler in production
if (isProd) {
  const navigationRoute = new NavigationRoute(serwist.precacheStrategy, {
    allowlist: [/^(?!\/__).*/],      // all paths except Nitro internals (/__...)
    denylist:  [/^\/_serverFn/],    // belt-and-suspenders: never precache server functions
  })
  serwist.registerCapture(navigationRoute)
}

6. SW registration — skipping dev, guarding SSR

The service worker is registered from the root route's shell component via a useSw hook. Two guards keep it clean: 

// src/hooks/use-sw.ts
export const useSw = () => {
  useEffect(() => {
    if (import.meta.env.DEV) return          // skip in dev entirely
    if (!('serviceWorker' in navigator)) return  // guard for unsupported browsers

    const serwist = new Serwist('/sw.js', { scope: '/', type: 'module' })
    serwist.register().catch(console.error)
  }, [])
}

The import.meta.env.DEV guard deserves explanation. The SW file does exist in dev (the buildStart hook creates it) and the browser would happily register it. But the dev SW uses NetworkOnly for everything, so registering it adds latency (all requests now go through a SW that immediately passes them through) with zero benefit. Skipping registration in dev means the SW is never in the critical path during active development.

The useEffect wrapper also implicitly handles the SSR boundary — useEffect never runs on the server, so navigator is never accessed during SSR. The explicit 'serviceWorker' in navigator check is a belt-and-suspenders guard for environments where navigator exists but the SW API doesn't (older browsers, some WebViews).

The hook is called once in RootDocument — the TanStack Start shell component that renders on every page load: 

// src/routes/__root.tsx
function RootDocument({ children }) {
  const { user } = Route.useRouteContext()
  useSw()  // registers the SW once, no-ops in dev and SSR

  return (
    <html lang='en' suppressHydrationWarning>
      <head><HeadContent /></head>
      <body>
        <ThemeProvider>
          <NavigationProgress />
          <Overlay />          // modal manager — always mounted
          {children}
          <Toaster />          // toast notifications
          <Scripts />          // TanStack Start's script injector
        </ThemeProvider>
      </body>
    </html>
  )
}

7. useIsOnline — why navigator.onLine isn't enough

navigator.onLine is notoriously unreliable. It returns true when the device has a network interface up — which includes a connected WiFi with no actual internet routing, a VPN tunnel with a broken remote endpoint, or a captive portal that hasn't been dismissed. A device "online" according to navigator.onLine may be completely unable to reach any real server.

The hook adds a real connectivity check on top: 

// src/hooks/use-is-online.ts
export function useIsOnline() {
  const [isOnline, setIsOnline] = useState(
    typeof navigator !== 'undefined' ? navigator.onLine : true
  )

  useEffect(() => {
    const updateStatus = async () => {
      if (!navigator.onLine) {
        setIsOnline(false)
        return
      }

      try {
        // Real probe: HEAD request to /favicon.ico
        // Cache-busted with ?t=timestamp to bypass the service worker cache
        const response = await fetch(`/favicon.ico?t=${Date.now()}`, {
          method: 'HEAD',
          mode:   'no-cors',
        })
        // With no-cors, response.ok will be false but response itself is truthy
        setIsOnline(response.ok || !!response)
      } catch {
        setIsOnline(false)
      }
    }

    window.addEventListener('online',  updateStatus)
    window.addEventListener('offline', updateStatus)
    updateStatus()  // run immediately on mount

    return () => {
      window.removeEventListener('online',  updateStatus)
      window.removeEventListener('offline', updateStatus)
    }
  }, [])

  return isOnline
}

The probe fetches /favicon.ico with a ?t=timestamp cache buster. The timestamp is critical: without it, the service worker's CacheFirst handler for images would serve the cached favicon even when offline, making the probe always succeed. The cache buster forces a new URL that has no cached entry, so the request always hits the network.

The mode: 'no-cors' flag means the response will be opaque — response.ok is always false for opaque responses, so the check is response.ok || !!response. An opaque response that doesn't throw is still evidence that the network is reachable. A thrown error means it isn't.

Where useIsOnline is used

  • › Login page — switches between AuthEngine.loginOnline and AuthEngine.loginOffline, changes the status badge and the form description
  • › Private route guard — determines whether to resolve identity from the server context or the local auth store
  • › Collection factory — every queryFn, onInsert, onUpdate, and onDelete checks navigator.onLine before attempting a server call

8. AppWrapper — the SSR hydration gate

TanStack Start is a full-stack SSR framework. The root beforeLoad runs on the server. Every route component is server-rendered before hydration. This creates a problem: a significant portion of the app — anything that touches navigator, sessionStorage, OPFS collections, or the BroadcastChannel — will throw if it runs on the server.

The standard approach is to scatter typeof window !== 'undefined' guards throughout the code. The problem with that: it's error-prone, invisible, and mixes environment concerns into business logic. The approach here is to push all of that behind a single, explicit gate: AppWrapper. 

// src/components/custom/app-wrapper.tsx
export function AppWrapper({ children }: { children: ReactNode }) {
  const [isClient, setIsClient] = useState(false)

  useEffect(() => {
    setIsClient(true)  // fires only in the browser, never on the server
  }, [])

  if (!isClient) return <Loading className='w-screen h-screen' />

  return children
}

It wraps the private layout route — the parent of every authenticated page: 

// src/routes/(private)/route.tsx
export const Route = createFileRoute('/(private)')({
  component: () => (
    <AppWrapper>
      <RouteComponent />  // ← only renders after hydration
    </AppWrapper>
  ),
})

Everything inside RouteComponent — useLiveQuery on OPFS collections, useIsOnline which accesses navigator, authStore.state which reads sessionStorage — runs only after isClient is true. The server renders the Loading spinner. The browser hydrates, useEffect fires, isClient flips, and the real content mounts.

What this gates

  • › TanStack DB collections (OPFS / IndexedDB)
  • › navigator.onLine access in useIsOnline
  • › sessionStorage reads for hotline assignment
  • › BroadcastChannel cross-tab sync
  • › Auth store reads that depend on client-only state

What it doesn't gate

  • › The root shell (RootDocument) — renders on both server and client
  • › Public routes (login page) — handled by their own SSR guards
  • › beforeLoad functions — those always run server-side

9. The root shell — wiring it all together

The TanStack Start root route ties the layers together. The beforeLoad function runs on the server on every request — it calls getAuthUser() to validate the Better Auth cookie and fetch the user from Postgres. If the request is offline (or the cookie is absent) it returns { user: undefined } without throwing. The private route guard picks up from there. 

// src/routes/__root.tsx
export const Route = createRootRouteWithContext<MyRouterContext>()({
  head: () => ({
    links: [
      { rel: 'manifest',         href: '/manifest.json'  }, // PWA installability
      { rel: 'apple-touch-icon', href: '/logo192.png'    }, // iOS home screen icon
    ],
  }),

  beforeLoad: async () => {
    try {
      const user = await getAuthUser()  // server-side: validates cookie + fetches profile
      setUser(user!)                     // idempotent — won't overwrite existing user
      return { user }
    } catch {
      return { user: undefined }         // offline or unauthenticated — don't throw
    }
  },

  shellComponent: RootDocument,
})

function RootDocument({ children }) {
  const { user } = Route.useRouteContext()
  useSw()  // SW registration — no-ops in dev and SSR

  useEffect(() => {
    if (user) setUser(user)  // client-side re-hydration of the auth store
  }, [user])

  return (
    <html>
      <head><HeadContent /></head>
      <body>
        <ThemeProvider>
          <NavigationProgress />  // top loading bar on route transitions
          <Overlay />             // modal manager singleton
          {children}              // route content (may be behind AppWrapper)
          <Toaster />             // toast notifications
          <Scripts />             // TanStack Start script injection
        </ThemeProvider>
      </body>
    </html>
  )
}

The setUser in useEffect might look redundant given it also runs in beforeLoad. It isn't: beforeLoad runs on the server, which means the result populates the route context but doesn't persist across a full client navigation. The useEffect re-applies the user to the reactive authStore on the client side after hydration. And because setUser is idempotent, if an offline login has already populated the store, this is a no-op.

11. Tradeoffs and known limitations

What this gets right

  • › Server functions are categorically excluded from all caching. There's no configuration that could accidentally cache a mutation response
  • › The prod/dev SW split is total — dev is NetworkOnly for everything, eliminating an entire class of "it works in dev, broken in prod" caching bugs
  • › The SSR boundary is explicit and centralized. One component gates all browser-only code — no scattered typeof window guards
  • › The connectivity probe verifies actual reachability, not just network interface status — the auth and data layers can trust the isOnline signal
  • › skipWaiting and clientsClaim mean a newly deployed SW takes effect immediately without a manual refresh — important for agents who keep the app open all day

Known limitations

  • › AppWrapper introduces a layout shift. On first load the server renders a spinner; after hydration the real content mounts. For a field tool this is acceptable — but it would need to be reconsidered for a content-heavy public site
  • › OPFS requires cross-origin isolation headers in production. Hosting platforms that strip or override custom headers (some CDN configurations, certain proxy setups) will silently break SQLite persistence. The failure mode is an empty collection on load, not a thrown error
  • › The connectivity probe adds latency to the initial mount. The HEAD request fires on every useEffect run. On a high-latency connection this creates a brief period where the UI doesn't know its online state yet
  • › skipWaiting can cause problems with multiple open tabs. When a new SW activates and claims all clients, any tab that has already loaded the old JS bundle may make requests that the new SW handles differently. For this app the risk is low but it's worth documenting
